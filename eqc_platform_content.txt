# EQC Asset Management Platform — Backend Application Plan
## Context
OTH Equipment Co. (Olympus Thailand Medical Division) currently manages medical device (endoscopy equipment) deployment, loaner, BOM, and inspection workflows across fragmented tools — MS Access, Excel, and Salesforce. The EQC Asset Management Platform replaces these with a single, enterprise-grade cloud system.
This plan covers the **backend application only** (separate from the React frontend). The backend is a Node.js/Express/TypeScript REST API deployed to Azure, with Azure SQL as the database. It serves the frontend app, inbound webhooks from Salesforce (via MuleSoft) and SAP, outbound alerts to MS Teams, and AI agents for autonomous monitoring.
**Why this plan:** We need a complete, file-level specification so any developer can pick it up and implement without ambiguity. The DB is Azure SQL (not PostgreSQL). All modules, middleware, flows, connectors, and processes are documented below.
---
## 1. Technology Stack
| Layer | Technology | Reason |
|---|---|---|
| Runtime | Node.js 20 LTS | Stable, wide Azure support |
| Framework | Express.js | Lightweight, well-understood |
| Language | TypeScript (strict) | Type safety across all modules |
| ORM | Prisma | Schema-first, works with Azure SQL (sqlserver provider) |
| Database | Azure SQL (SQL Server) | Enterprise managed DB, HA zone-redundant |
| Job Queue | BullMQ + Azure Cache for Redis | Cron jobs, async email/Teams tasks |
| Auth | JWT (RS256) + Azure AD OAuth2 | SSO + fallback email/password |
| PDF Generation | PDFKit | Server-side A4 transport docs |
| QR Code | qrcode library | Embedded in transport PDFs |
| Email | Nodemailer + Azure Communication Services / SendGrid | Print queue, escalations |
| Blob Storage | Azure Blob Storage SDK | PDFs, signed copies, reports |
| AI | Azure OpenAI (GPT-4o mini) | 6 AI agents for autonomous ops |
| Validation | Zod | Schema validation on all requests |
| Logging | Winston + Azure Application Insights | Structured logs, APM |
| Security | Helmet, express-rate-limit, CORS | OWASP hardening |
| Testing | Jest + Supertest | Unit + integration tests |
| Containerization | Docker + Azure Container Registry | Consistent builds |
| Orchestration | Azure Kubernetes Service (AKS) | Autoscaling 2–10 nodes |
| CI/CD | Azure DevOps Pipelines | Build, test, push, deploy |
| Secrets | Azure Key Vault | All credentials, never in code |

```
│   │   ├── validate.middleware.ts  ← ZOD VALIDATION. Exports `validate(schema: ZodSchema)` which validates `req.body`, `req.query`, and `req.params` against the provided Zod schema. Returns 400 with field-level error details on failure. All route bodies must have a corresponding Zod schema file.
```

- `DeviceDeployment`: index on `(status)`, `(start_date, end_date)`, `(device_id, status)`
- `Asset`: index on `(serial_number)`, `(status)`, `(model_code)`
- `AuditLog`: index on `(entity_type, entity_id)`, `(timestamp DESC)`
- `DispatchDocument`: index on `(status, created_at)`, `(deployment_id)`

### Concurrent Booking Prevention (Azure SQL approach)
- Use Prisma's `$transaction()` with `serializable` isolation level
- Within transaction: check for overlapping active deployments on same device before creating new one
- Azure SQL handles pessimistic locking at the row level within the serializable transaction

## 4. All Flows & Processes
### 4.1 Authentication Flow
**Azure AD SSO (Primary):**
1. Frontend redirects user to Azure AD OAuth2 authorize URL (constructed by frontend)
2. User authenticates with Olympus corporate credentials in Azure AD
3. Azure AD redirects to `/api/auth/sso/azure?code=...`
4. Backend: exchanges code for Azure AD access token → fetches user profile (email, name, azureAdObjectId)
5. Backend: calls `user.service.upsertFromAzureAD()` — creates user if first login, updates if existing
6. Backend: calls `auth.service.issueTokenPair(user)` — creates JWT access token (15min) + refresh token (7 days)
7. Refresh token hash stored in DB against user record
8. Access token returned in response body; refresh token set as HttpOnly, SameSite=Strict cookie
9. Frontend stores access token in memory (NOT localStorage)
**Email/Password (Fallback):**
1. `POST /api/auth/login` with `{ email, password }`
2. Service finds user by email, checks isActive
3. bcrypt.compare(password, user.passwordHash) — constant-time comparison
4. If match: issue token pair same as SSO flow
5. If fail: return 401 (same generic message regardless of whether email exists — prevents enumeration)
**Token Refresh:**
1. Frontend intercepts 401 response from any API call
2. Frontend calls `POST /api/auth/refresh` (refresh token sent automatically via cookie)
3. Backend: reads HttpOnly cookie, verifies refresh token hash matches DB record, checks expiry
4. Issues new access token (sliding window reuse detection: if same refresh token used twice, revoke all)
5. Returns new access token
6. Frontend retries original request with new token
**Logout:**
1. `POST /api/auth/logout`
2. Backend: deletes refresh token hash from DB
3. Backend: clears the HttpOnly cookie
4. Frontend: clears in-memory access token

### 4.2 Demo Deployment Full Lifecycle
```
SFDC (Sales Rep) → MuleSoft → POST /api/webhooks/mulesoft/new-request
  │ Validates HMAC signature
  │ Checks idempotency (sfdc_request_id)
  │ Creates DeviceDeployment (status: PENDING_EQC)
  │ Upserts Account if new
  │ Returns 200
EQC Manager → PATCH /api/deployments/:id/approve
  │ requireRole(MANAGER, ADMIN)
  │ Validates transition PENDING_EQC → IN_PREPARATION
  │ Checks asset availability (no overlapping active deployments)
  │ Updates deployment status
  │ Posts Teams card to #eqc-ops-alerts: "New deployment ready for preparation"
  │ Returns 200
EQC Operator → POST /api/bom/snapshots (creates frozen BOM)
  │ Reads all BOMComponents for asset's model_code + set
  │ Serializes to JSON in BOMSnapshot table
  │ Links to deployment
EQC Operator → POST /api/bom/validate-packing { snapshotId, packedItemIds[] }
  │ Checks all REQUIRED items in packedItemIds
  │ Returns { isComplete, missingItems }
EQC Operator → POST /api/dispatch/documents { deploymentId }
  │ Calls validatePacking internally → if incomplete: returns 409 DISPATCH_BLOCKED
  │ Generates PDF (PDFKit): A4 with BOM list + QR code
  │ Uploads PDF to Azure Blob (transport-docs container)
  │ Creates DispatchDocument record
  │ Transitions deployment → IN_TRANSIT
  │ Transitions asset → DEMO
  │ Posts Teams card to #eqc-ops-alerts: "Dispatch ready"
  │ Queues email job to warehouse printer
  │ Returns { docId, pdfUrl, qrPayload }
[Hospital Receives Device — manual confirmation or QR scan]
Hospital Returns Device:
  → EQC Operator: PATCH /api/deployments/:id/status { status: RETURNED }
    │ Transitions deployment → RETURNED
    │ Asset remains in DEMO status until inspection complete
EQC Operator → POST /api/inspections { deploymentId }
  │ Reads frozen BOMSnapshot
  │ Creates one InspectionRecord row per component (result: null)
  │ Returns inspection with component list
EQC Operator → PATCH /api/inspections/:id/items/:componentId { result, conditionNote }
  │ Updates InspectionRecord.result
  │ If result = FAIL or MISSING:
  │   → Generates repairCaseId (RC-YYYYMMDD-XXX)
  │   → Updates InspectionRecord.repair_case_id
  │   → Posts CRIT Teams card to #asset-defects
EQC Operator → POST /api/inspections/:id/complete
  │ Validates all components assessed
  │ If all PASS: Asset → AVAILABLE, updates last_sterilized_at, deployment → COMPLETED
  │ If any FAIL/MISSING: Asset → IN_REPAIR, deployment → COMPLETED
  │ Writes audit log entry
```

### 4.3 BOM Dispatch Block Enforcement
This is a hard, server-side constraint — cannot be bypassed by the frontend.
```
dispatch.service.generateDocument(deploymentId, userId):
  1. Load BOMSnapshot for deployment (throws 404 if not frozen yet)
  2. Parse snapshot.components JSON → get all components where item_type = 'REQUIRED'
  3. Compare with packedItemIds[] passed in request body
  4. If any REQUIRED component not in packedItemIds:
     → Throw AppError.dispatchBlocked(missingItems)
     → HTTP 409, errorCode: DISPATCH_BLOCKED
     → Response includes missingItems[] array for UI display
  5. If all REQUIRED present:
     → Proceed with PDF generation
```

### 4.4 Overdue Alert Pipeline (Automated)
```
Every 6 hours via BullMQ cron:
  overdueAlert.processor.ts runs:
  1. Query DB: SELECT deployments WHERE status = 'DELIVERED' AND end_date < NOW()
  2. For each overdue deployment:
     a. Calculate daysOverdue
     b. Classify severity:
        - 1-3 days: MEDIUM (log only)
        - 4-7 days: HIGH → Teams alert to #loaner-overdue or #demo-alerts
        - 8-14 days: CRIT → Teams alert + manager email
        - >14 days: URGENT → Teams alert + manager email + executive escalation
     c. Check if already alerted at this severity (avoid duplicates via alert_failures or a separate alert_sent flag)
     d. If needs alert: call overdueClassifier.agent.ts (AI) to draft escalation message
     e. Post Teams Adaptive Card with Acknowledge/Escalate action buttons
     f. Update record to mark alert sent
```

### 4.5 MuleSoft Integration Pipeline
```
Inbound (SFDC → Backend):
  MuleSoft (Azure API Management gateway) → POST /api/webhooks/mulesoft/new-request
  Headers: X-Mulesoft-Signature: hmac-sha256({rawBody}, MULESOFT_WEBHOOK_SECRET)
  webhook.validator.ts:
    1. Read raw request body (express.raw() used on webhook routes — NOT express.json())
    2. Compute HMAC-SHA256 of raw body
    3. Compare with header value (constant-time comparison via crypto.timingSafeEqual)
    4. If mismatch: return 401 immediately, log attempt to Azure Monitor
  On valid signature:
    1. Parse JSON body
    2. Check idempotency: SELECT FROM DeviceDeployment WHERE sfdc_request_id = req.body.id
       → If exists: return 200 with existing record (idempotent)
       → If new: create deployment
    3. All processing inside try-catch
    4. Return 200 regardless of processing success (to prevent MuleSoft retry storms)
    5. Log errors to Azure Monitor for investigation
Outbound (Backend → SFDC via MuleSoft):
  (Phase 2 — bidirectional status sync)
  When deployment status changes: POST to MuleSoft outbound endpoint with status update
```

### 4.6 Teams Integration Pipeline
```
Outbound alerts (Backend → Teams):
  teams.service.ts.postDispatchCard(deployment, docUrl):

  2. POST to channel webhook URL from config
```

## 6. Middleware Stack (Execution Order)
For every authenticated API request, middleware executes in this order:
```
1. Helmet (security headers)
2. CORS (restrict to frontend origin)
3. express.json() (parse request body, limit 1mb)
4. requestLogger.middleware (log request)
5. rateLimiter.middleware (check rate limit)
6. [Route match]
7. auth.middleware (verify JWT, attach req.user) ← skipped for /auth/* and /webhooks/*
8. rbac.middleware (check role against route requirement) ← where applied
9. validate.middleware (Zod schema check on body/query/params) ← where applied
10. asyncHandler (wraps next function)
11. Route Handler (controller method)
12. audit.middleware (write audit log non-blocking) ← where applied to state-changing routes
13. errorHandler.middleware (catches any error from above, formats response)
```
