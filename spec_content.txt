Document Purpose
This document provides the complete technical specification for the EQC backend application — a Node.js/Express/TypeScript REST API deployed on Azure Kubernetes Service (AKS), using Azure SQL as the primary database. It covers every file and its purpose, all API routes, authentication and authorization flows, integration connectors, background job processors, AI agents, database schema, and verification procedures.
Table of Contents
1. Executive Summary
The EQC Asset Management Platform is an enterprise-grade cloud system replacing fragmented MS Access, Excel, and Salesforce workflows used to manage medical endoscopy device deployments across hospitals in Thailand. This document specifies the backend application exclusively — a standalone Node.js/Express/TypeScript REST API running on Azure Kubernetes Service (AKS) with Azure SQL as the primary database.
The backend serves the React frontend (a separate application), processes inbound webhooks from Salesforce via MuleSoft and from SAP for daily asset master synchronization, sends outbound Adaptive Card alerts to six MS Teams channels, generates A4 PDF transport documents stored in Azure Blob Storage, and runs six AI agents for autonomous operations monitoring powered by Azure OpenAI GPT-4o mini.
The system manages the full lifecycle of medical endoscopy device sets: Demo deployments to hospital surgical departments, Loaner deployments triggered by repair requests, BOM (Bill of Materials) packing and dispatch with hard server-enforced blocking on incomplete packing, return inspection with per-component pass/fail assessment and automatic repair case creation, and a complete append-only audit trail for compliance.
Key architectural principles:
State machines enforced in the service layer — invalid transitions are rejected with 409
BOM dispatch block is server-side only — cannot be bypassed by the frontend under any circumstances
BOM snapshots are frozen at dispatch time — inspections use the historical snapshot, not live BOM data
All webhook processing is idempotent — duplicate calls return existing records
Audit logging is non-blocking — never delays HTTP responses
All secrets managed via Azure Key Vault — never stored in code or environment files

4. Process Flows
4.1 Azure AD SSO Authentication Flow
User clicks 'Sign in with Olympus AD' on the frontend application.
Frontend redirects to Azure AD authorize URL with client_id, redirect_uri, response_type=code, scope=openid email profile.
User authenticates using Olympus corporate credentials inside Azure AD.
Azure AD redirects to GET /api/auth/sso/azure?code={auth_code} on the backend.
auth.service.azureCallback(code): exchanges authorization code for Azure AD access token via POST to Azure AD token endpoint.
Backend fetches user profile from Microsoft Graph API using the Azure AD access token.
Calls user.service.upsertFromAzureAD(azureAdObjectId, profile): creates the user record on first login, updates name/email on subsequent logins.
Calls auth.service.issueTokenPair(user): signs a JWT access token (15-minute expiry) and generates a cryptographically random refresh token.
Stores the SHA-256 hash of the refresh token in the User.refreshTokenHash field in the database.
Sets an HttpOnly, SameSite=Strict, Secure cookie containing the refresh token value.
Returns the access token in the JSON response body.
Frontend stores the access token in memory only — never in localStorage or sessionStorage.
All subsequent API calls include Authorization: Bearer {access_token} header.
4.2 Demo Deployment Full Lifecycle
Sales rep creates a Demo Request in Salesforce for a hospital visit, specifying device model, dates, and account.
MuleSoft detects the new Salesforce record and calls POST /api/webhooks/mulesoft/new-request with HMAC-SHA256 signature.
Backend validates HMAC signature, checks sfdc_request_id for idempotency, creates DeviceDeployment (status: PENDING_EQC), and upserts the Account record.
EQC Manager logs in to the web app, sees the pending request in the service dashboard.
Manager calls PATCH /api/deployments/{id}/approve — system validates no date conflicts, transitions status to IN_PREPARATION, posts Teams Adaptive Card to #eqc-ops-alerts.
EQC Operator calls POST /api/bom/snapshots to freeze the BOM for this deployment — serializes all BOM components to JSON in BOMSnapshot table.
Operator physically picks components, then calls POST /api/bom/validate-packing to verify all REQUIRED items are present.
Operator calls POST /api/dispatch/documents — system checks dispatch block (returns 409 if any REQUIRED item missing), generates A4 PDF with embedded QR code, uploads to Azure Blob, transitions deployment to IN_TRANSIT, queues print email to warehouse printer.
Device is physically dispatched to the hospital. Hospital receives the device and signs the paper transport document.
EQC Operator scans the signed copy and calls POST /api/dispatch/documents/{id}/sign with multipart upload — stored in Azure Blob signed-copies container, DispatchDocument.status set to SIGNED.
Hospital returns device. EQC Operator calls PATCH /api/deployments/{id}/status with status: RETURNED.
EQC Operator calls POST /api/inspections to create inspection records (one per BOM component). Records each component result via PATCH on each item. Any FAIL or MISSING auto-creates a repair case (RC-YYYYMMDD-XXX) and posts CRIT Teams alert to #asset-defects. POST /api/inspections/{id}/complete finalizes: all PASS → asset to AVAILABLE; any FAIL → asset to IN_REPAIR; deployment to COMPLETED.
4.3 BOM Dispatch Block Enforcement
The dispatch block is a hard server-side constraint. It cannot be bypassed by the frontend under any circumstances.
EQC Operator calls POST /api/dispatch/documents with { deploymentId }.
dispatch.service.generateDocument() first calls bom.service.validatePacking(snapshotId, packedItemIds[]).
validatePacking reads all components from the frozen BOMSnapshot where item_type = REQUIRED.
Checks which required component IDs are present in the packedItemIds array provided by the client.
If any REQUIRED component is absent: returns { isComplete: false, missingItems: [{componentId, componentName, requiredQty}...] }.
dispatch.service receives isComplete = false: throws AppError.dispatchBlocked(missingItems).
errorHandler.middleware catches this AppError: returns HTTP 409 with body { success: false, error: { code: 'DISPATCH_BLOCKED', message: 'Cannot dispatch — missing required BOM items', details: { missingItems: [...] } } }.
Frontend displays the missing items list to the EQC operator.
Operator physically locates the missing items, adds them, and re-calls validate-packing.
Once all REQUIRED items confirmed as packed: POST /api/dispatch/documents proceeds to PDF generation.
4.4 MuleSoft Inbound Integration Flow
Salesforce event triggers MuleSoft Experience API.
MuleSoft computes HMAC-SHA256 of the raw request body using the shared secret and adds it as X-Mulesoft-Signature header.
MuleSoft calls POST /api/webhooks/mulesoft/new-request with raw JSON body.
webhook.routes.ts applies express.raw() middleware on this route — preserves raw body bytes for signature computation.
webhook.validator.ts reads X-Mulesoft-Signature header, computes HMAC-SHA256(rawBody, MULESOFT_WEBHOOK_SECRET) using Node.js crypto module.
Compares computed signature with the header value using crypto.timingSafeEqual() to prevent timing-based attacks.
If signatures do not match: immediately returns 401 Unauthorized and logs the attempt to Azure Monitor.
If valid: parses JSON body (JSON.parse(rawBody)).
Checks DeviceDeployment table for existing record where sfdc_request_id = body.id (idempotency check).
If duplicate: returns 200 with existing record to prevent MuleSoft from creating duplicate deployments on retry.
If new: creates deployment, upserts account, returns 201.
All errors inside processing are caught and logged — always returns 200 to prevent MuleSoft retry storms on application-level failures.
4.5 Automated Overdue Alert Pipeline
BullMQ cron job fires every 6 hours via overdueAlertQueue.
overdueAlert.processor.ts queries: DeviceDeployment WHERE status = DELIVERED AND end_date < NOW().
For each overdue deployment: calculates daysOverdue = Math.floor((now - endDate) / 86400000).
Classifies severity: 1-3 days = MEDIUM (log only), 4-7 days = HIGH, 8-14 days = CRIT, >14 days = URGENT.
Reads alert_sent_at and last_alert_severity from the deployment record — skips if this severity level was already alerted.
For new severity: calls overdueClassifier.agent.ts with full deployment context.
overdueClassifier.agent.ts calls openai.service.complete() with deployment data, receives { severity, suggestedAction, emailDraft }.
teams.service.postOverdueAlert(deployment, severity, emailDraft) posts Adaptive Card to #loaner-overdue or #demo-alerts with device details, days overdue, AI-drafted message, and action buttons.
Updates deployment record: alert_sent_at = now, last_alert_severity = severity.
When action button is clicked in Teams: Teams POST to /api/webhooks/teams, teams.controller routes the acknowledgment or escalation action.
4.6 MS Teams Adaptive Card Integration
Outbound (Backend to Teams)
teams.service constructs an Adaptive Card JSON payload and POSTs it to the appropriate Teams channel webhook URL from config. Retry logic: 3 attempts with exponential backoff (500ms, 1000ms, 2000ms). On complete failure: writes to AlertFailure table for manual retry via admin dashboard.
Inbound (Teams Card Actions to Backend)
When a user clicks Acknowledge, Escalate, or Review on a Teams card, Teams sends POST to /api/webhooks/teams with the card action payload. teams.controller.handleCardAction() validates the Teams HMAC signature, parses the action type, and routes to the appropriate service method (mark alert acknowledged, create manager escalation, etc.). Returns 200 to Teams immediately.

5.2 JWT Token Structure
Access Token Payload:
{ sub: "userId (UUID)", email: "user@olympus.th", role: "EQC", azureAdObjectId: "...", iat: 1234567890, exp: 1234568790 }
Access token expiry: 15 minutes (900 seconds)
Refresh token expiry: 7 days (604800 seconds) — stored as HttpOnly cookie
Refresh token stored as SHA-256 hash in DB — raw value never persisted
On token reuse detection (same refresh token used twice): all sessions revoked
5.3 RBAC Middleware Implementation
The requireRole factory is applied per-route:
router.patch("/approve", requireRole("MANAGER", "ADMIN"), asyncHandler(controller.approve))
On insufficient role: returns HTTP 403 with { success: false, error: { code: 'INSUFFICIENT_PERMISSIONS', message: '...' } }.
5.4 Middleware Execution Order
For every authenticated API request, middleware executes in this fixed order:
Helmet (sets security headers: CSP, HSTS, X-Frame-Options, etc.)
CORS (restrict to FRONTEND_ORIGIN from config)
express.json() (parse body, 1MB limit)
requestLogger.middleware (log every request)
rateLimiter.middleware (check per-route rate limits)
[Route match]
auth.middleware (verify JWT, attach req.user) — skipped for /auth/* and /webhooks/*
rbac.middleware (check role against route requirement) — where applied
validate.middleware (Zod schema check on body/query/params) — where applied
asyncHandler (wraps next function in try-catch)
Route Handler (controller method)
audit.middleware (write audit log non-blocking via setImmediate) — on state-changing routes
errorHandler.middleware (catches any thrown error, formats and sends response)
6. Integration Connectors
6.1 Salesforce / MuleSoft Connector
Direction: Bi-directional
Method: HMAC-SHA256 signed webhooks via MuleSoft Experience API
Inbound endpoints: POST /api/webhooks/mulesoft/new-request, POST /api/webhooks/mulesoft/status-update, POST /api/webhooks/mulesoft/asset-sync
Security: HMAC-SHA256 signature validation on every call using crypto.timingSafeEqual()
Idempotency: sfdc_request_id checked before creating any record
Error handling: always return 200 to prevent MuleSoft retry storms; log errors to Azure Monitor
6.2 SAP S/4HANA Connector
Direction: Read-only inbound synchronization
Method: Daily batch via MuleSoft asset-sync webhook endpoint
Data synchronized: Material Master (BOM component part numbers), Equipment Asset (SAP asset numbers)
Schedule: Daily at 02:00 UTC — processes via webhook.service.processAssetSyncBatch()
All SAP data is reference-only — no writes back to SAP in current phase
6.3 MS Teams Connector
Direction: Outbound alerts + inbound action callbacks
Channels (6): #eqc-ops-alerts, #loaner-overdue, #demo-alerts, #asset-defects, #inventory-critical, #exec-summary
Method: Incoming Webhooks (outbound) + callback endpoint /api/webhooks/teams (inbound)
Retry: 3 attempts with exponential backoff on outbound failures
Failure handling: AlertFailure table for manual retry via admin dashboard
6.4 Azure OpenAI Connector
Direction: Outbound (backend calls Azure OpenAI REST API)
Model: GPT-4o mini (cost-optimized for high-volume inference)
Token budgets: 300-2000 tokens per inference depending on agent type
Retry: 3x on HTTP 429 (rate limit) and 503 (unavailable) with exponential backoff
Usage: 6 AI agents running as BullMQ background workers — never in request path
6.5 Azure Blob Storage Connector
Containers: transport-docs (dispatch PDFs), signed-copies (signed return documents), audit-reports (compliance reports)
Access pattern: Private storage — SAS tokens generated on demand with 1-hour expiry
SAS tokens are NOT stored in the database — generated fresh when user requests a download
Blob paths stored in DB (not full URLs) to allow SAS generation for any environment
6.6 Email / SMTP Connector
Provider: Azure Communication Services SMTP or SendGrid as fallback
All email sending is asynchronous — queued via BullMQ emailQueue, never inline in request path
Use cases: warehouse printer notifications (with PDF attachment), manager escalations, scheduled report distribution

7.1 Critical Performance Indexes
The following indexes are defined in schema.prisma for Azure SQL query performance:
DeviceDeployment: index on (status), (start_date, end_date), (device_id, status)
Asset: index on (serial_number), (status), (model_code)
AuditLog: index on (entity_type, entity_id), (timestamp DESC)
DispatchDocument: index on (status, created_at), (deployment_id)
Overdue scan: filtered index on DeviceDeployment(status, end_date) WHERE status = 'DELIVERED'
7.2 Concurrent Booking Prevention
To prevent two users from booking the same device for overlapping dates simultaneously, deployment creation uses Prisma's $transaction() with serializable isolation level. Within the transaction, the system queries for any active deployment on the same device with overlapping date ranges before creating the new record. Azure SQL handles row-level locking within the serializable transaction, ensuring only one booking succeeds in a race condition.

9. Verification Plan
After implementation, verify end-to-end correctness using these 8 steps:
Schema Migrations: Run prisma migrate dev against Azure SQL Edge (local Docker). Confirm all 13 tables created without errors. Verify all indexes exist using Azure Data Studio or SSMS.
Authentication: POST /api/auth/login with test user credentials → receive JWT access token + cookie. Call GET /api/auth/me → verify user object returned. Call any protected route without token → confirm HTTP 401. Call with expired token → confirm 401 with TOKEN_EXPIRED code.
RBAC: Call GET /api/users (ADMIN only) with a SALES-role token → confirm HTTP 403 INSUFFICIENT_PERMISSIONS. Repeat with ADMIN token → confirm HTTP 200.
BOM Dispatch Block: Create test deployment, freeze BOM snapshot, call POST /api/dispatch/documents with an incomplete packed items list → confirm HTTP 409 DISPATCH_BLOCKED with missingItems array in response. Add the missing items and call again → confirm HTTP 200 with a pdfUrl and docId.
State Machine Enforcement: Try transitioning an asset from AVAILABLE directly to COMPLETED → confirm HTTP 409 conflict. Try a valid transition (AVAILABLE → DEMO) → confirm HTTP 200. Try to transition a RETIRED asset to any status → confirm HTTP 409.
MuleSoft Webhook Idempotency: POST /api/webhooks/mulesoft/new-request with a valid HMAC signature and a new sfdc_request_id → confirm HTTP 201 and new deployment created. Send the exact same request again → confirm HTTP 200 returned with the same existing deployment record (no duplicate created). Send the same request with an invalid HMAC signature → confirm HTTP 401.
Teams Alert (mocked): Trigger the overdue alert processor manually via /api/admin/agents/overdue-alert/trigger. Confirm in logs that teams.service.postOverdueAlert was called with the correct deployment data. In non-production environment this should use the mockTeamsService which resolves without making HTTP calls.
Integration Test Suite: Run npm test -- --testPathPattern=integration from the project root. All integration tests must pass with 100% success rate before any deployment to staging.
